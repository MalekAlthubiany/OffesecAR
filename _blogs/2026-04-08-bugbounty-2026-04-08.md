## المشكلة: عندما تنهار الحدود بين العملاء

تعتمد التطبيقات متعددة المستأجرين (Multi-Tenant) على مبدأ أساسي: عزل بيانات كل عميل عن الآخر بشكل صارم. في Laravel، يستخدم المطورون آلية Global Scopes لتطبيق هذا العزل تلقائياً على جميع الاستعلامات، حيث تضيف شرط `WHERE tenant_id = X` لكل query دون الحاجة لتكراره يدوياً.

لكن ما يحدث عندما تستخدم `orWhere()` في استعلامك؟ هنا يكمن الفخ.

الباحث الأمني اكتشف أن سلسلة استعلامات بسيطة مثل:

```php
User::where('status', 'active')
    ->orWhere('role', 'admin')
    ->get();
```

قد تنتج SQL كهذا:

```sql
SELECT * FROM users 
WHERE tenant_id = 5 
AND status = 'active' 
OR role = 'admin';
```

بسبب أولوية العمليات في SQL، هذا الاستعلام يُفسّر كـ:

```sql
(tenant_id = 5 AND status = 'active') OR (role = 'admin')
```

النتيجة؟ سيحصل المستخدم على جميع المسؤولين من **كل** المستأجرين، وليس فقط مستأجره!

## الآلية التقنية: كيف ينكسر العزل

المشكلة تنبع من كيفية عمل Global Scopes في Laravel. عندما تُعرّف Global Scope، يضيف Laravel الشرط في بداية WHERE clause. لكن عند استخدام `orWhere()` دون تجميع منطقي، تنكسر أولوية العمليات.

السيناريو الخطير يحدث عندما:

1. **يطبق Global Scope** شرط `tenant_id` أولاً
2. **يضيف المطور** شروط `WHERE` إضافية
3. **يستخدم `orWhere()`** دون closure
4. **ينتج SQL** يسمح بتجاوز شرط المستأجر

الحل الصحيح هو تجميع الشروط داخل closure:

```php
User::where(function ($query) {
    $query->where('status', 'active')
          ->orWhere('role', 'admin');
})->get();
```

هذا ينتج SQL آمن:

```sql
SELECT * FROM users 
WHERE tenant_id = 5 
AND (status = 'active' OR role = 'admin');
```

## خطوات الاكتشاف والاستغلال

الباحث اتبع منهجية منظمة:

1. **تحليل كود Laravel** الخاص بتطبيق متعدد المستأجرين
2. **تحديد نقاط استخدام orWhere** في Queries مع Global Scopes
3. **اختبار الاستعلامات** عبر SQL logging لرؤية الـ query الفعلي
4. **التحقق من تجاوز العزل** بالوصول لبيانات مستأجرين آخرين
5. **توثيق السيناريوهات** المختلفة التي تسبب المشكلة

الثغرة لا تقتصر على `orWhere()` فقط، بل تشمل أيضاً:

- استخدام `whereIn()` مع `orWhere()`
- Subqueries معقدة مع عمليات OR
- Eager loading relationships مع شروط OR

## التأثير الأمني: من IDOR إلى Data Breach كامل

هذه الثغرة تُصنّف كـ **Broken Access Control** وقد تؤدي إلى:

- **Horizontal Privilege Escalation**: الوصول لبيانات مستأجرين آخرين
- **Data Breach واسع النطاق**: تسريب قواعد بيانات عملاء بالكامل
- **انتهاك الخصوصية والامتثال**: خرق GDPR وقوانين حماية البيانات
- **فقدان الثقة**: تدمير سمعة الشركة كمزود SaaS

الخطورة تزداد في:

- تطبيقات SaaS الكبيرة
- أنظمة ERP و CRM متعددة المستأجرين
- منصات التجارة الإلكترونية
- أنظمة إدارة المحتوى متعددة العملاء

## دروس لصائدي الثغرات

### 1. افهم الـ ORM بعمق
لا تكتفِ بمعرفة كيفية استخدام Eloquent، بل افهم كيف يُترجم إلى SQL. استخدم `toSql()` أو `DB::listen()` لرؤية الاستعلامات الفعلية.

### 2. ابحث في الـ Logic Bugs
ليست كل الثغرات XSS أو SQLi. Logic bugs في Multi-tenancy تكون أحياناً أخطر وأعلى مكافأة.

### 3. اقرأ الكود المصدري
تطبيقات Laravel غالباً open source أو لديك access للكود في برامج Bug Bounty الخاصة. راجع Models، Scopes، والـ Queries.

### 4. اختبر Edge Cases
المطورون يختبرون السيناريوهات الطبيعية. أنت ابحث عن:
- استخدامات OR في queries
- Subqueries معقدة
- Relations مع custom scopes

### 5. وثّق بدقة
اشرح الـ root cause، ليس فقط الـ impact. أظهر الـ SQL query قبل وبعد، هذا يُقنع الفريق الأمني بسرعة.

## الحماية والتوصيات

للمطورين:

- **دائماً استخدم closures** مع orWhere في تطبيقات Multi-tenant
- **اكتب Unit Tests** تتحقق من عزل البيانات
- **استخدم Package مثل** `spatie/laravel-multitenancy` للحماية الإضافية
- **راجع كل query** يستخدم OR في كود review

لصائدي الثغرات:

- **فحص تطبيقات Laravel** متعددة المستأجرين بحثاً عن هذا النمط
- **التركيز على APIs** التي تقبل معاملات تصفية معقدة
- **اختبار Webhooks وBackend Jobs** التي قد تتجاهل Scopes
